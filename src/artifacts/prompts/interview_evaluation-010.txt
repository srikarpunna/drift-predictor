The core challenge is: you can't do a synchronous distributed consensus check at 500k RPS without blowing your latency budget, so you need to accept some eventual consistency in your counting. Here's how I'd approach it. I'd use a sliding window counter implemented in Redis Cluster — Redis single-node can handle about 100k ops/second on commodity hardware, so for 500k RPS I'd shard by user or entity ID across five nodes minimum with consistent hashing. Each rate check is a Lua script — atomically execute INCR, check against limit, and set TTL in one round-trip. Lua runs server-side in Redis, so you avoid a network roundtrip between increment and check. That keeps you well within 5ms at the rate check layer if your application servers are co-located regionally with Redis.

[00:06] Farida (Interviewer): What happens during a Redis failover in that architecture? What's the failure mode?

[00:07] Yuki (Candidate): During failover in Redis Sentinel or Redis Cluster, you have a leader election window — typically two to ten seconds depending on your heartbeat configuration. During that window, rate checks either fail open or fail closed depending on your policy. Failing open means requests pass through unchecked — you accept some abuse risk for those seconds. Failing closed means you drop legitimate traffic. For most rate-limiting use cases, fail-open is the better default: the cost of a few seconds of unthrottled traffic is lower than dropping legitimate customer requests. What you want to avoid is indefinite fail-open, so you'd want a maximum circuit-breaker timeout — if the Redis cluster is unreachable for more than 30 seconds, you'd route to a degraded mode that uses in-process rate limiting with a tighter limit as a safety backstop.

[00:11] Farida (Interviewer): How would you handle multi-region? You have users hitting different regions but their rate limit is global.

[00:12] Yuki (Candidate): Cross-region rate limiting is a hard problem because cross-region latency — say 80ms for US-East to EU-West — immediately blows your 5ms budget if you do a synchronous cross-region check. There are a few approaches. Option one: regional rate limits with a global envelope. Each region enforces a per-region limit — say 60% of the global limit — and a background sync job reconciles total usage across regions every few seconds. This is eventually consistent but keeps per-check latency local. Option two: token bucket with async replenishment. Each region has a local token bucket. A global coordinator periodically redistributes tokens based on observed usage patterns. This works well when traffic distribution across regions is predictable. Option three: approximate counting with CRDTs — conflict-free replicated data types that allow concurrent increments across regions and converge without coordination. Redis doesn't natively support CRDTs but you can implement a G-counter on top of it. The tradeoff is that you'll over-allow briefly during the convergence window. I'd use option one for most cases — it's predictable, debuggable, and the math is clear.

[00:19] Yuki (Candidate): This is a real footgun in Redis. Lua scripts block the entire Redis event loop while they execute — Redis is single-threaded for command processing. I've seen this in production: someone added a debug logging call inside a rate-limit Lua script that did a string concatenation in a hot path, and p99 latency on that Redis instance jumped from 1ms to 40ms under load. The mitigation is: Lua scripts for rate limiting must be kept minimal — INCR and comparison only, nothing else — and you should have a latency SLO on Lua script execution itself, monitored separately from overall Redis latency. [00:35] Yuki (Candidate): I approach it in layers. First, narrow the scope: is the latency regression at the client, the network, the rate-limit service itself, or a downstream dependency? I'd add temporary high-frequency metrics at each layer and bisect until the regression is isolated.

[00:40] Farida (Interviewer): What's one architectural decision in this kind of system that looks correct but commonly introduces subtle bugs at scale?

[00:41] Yuki (Candidate): Using wall clock time for sliding window boundaries without accounting for clock skew across nodes. If your rate limit window is "the last 60 seconds" and you're computing that from each app server's local clock, clock skew of even 200 milliseconds can cause windows to open slightly early or close slightly late on different nodes. Under normal conditions it doesn't matter. At high RPS with a tight limit, 200ms of window drift means a user can sneak in extra requests between nodes that disagree on where the window boundary is. The fix is to compute window boundaries from a single time source — Redis server time via the TIME command — rather than app server clocks. It adds one extra Redis call but eliminates the skew. I've seen this bite a team in a fraud prevention context where they needed tight per-second limits and kept seeing limit violations they couldn't explain.
