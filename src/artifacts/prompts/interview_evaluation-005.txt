Events come in through Kafka, which acts as the durable buffer. Two paths off Kafka: a streaming path for real-time dashboards, and a batch path for the warehouse.

For the streaming path I'd use Flink or Spark Structured Streaming to process events and write aggregated metrics — clicks per page, active users per minute — into a fast query store, maybe ClickHouse or Redis. Dashboard latency would be under ten seconds.

For the batch path, events in Kafka get consumed by a batch job every hour, written to S3 as Parquet, then loaded into a data warehouse — Snowflake or BigQuery. Analysts get full event history with high-fidelity queries but up to an hour of latency.

[00:06] Cleo (Interviewer): Why Kafka specifically over something like Kinesis?

[00:07] Ravi (Candidate): Kafka gives you more control — partition count, retention policy, consumer group behavior are all configurable. Kinesis is easier to manage if you're already on AWS and have lower operational overhead, but the shard limit and pricing model get painful at scale. At 50K events per second that's roughly 50MB/s assuming 1KB average event size. Kafka handles that easily with a 10-partition topic and reasonable replication. I'd choose Kafka if the team has the operational maturity to run it; Kinesis if they want a managed service with less tuning.

[00:09] Cleo (Interviewer): How do you handle late-arriving events in the streaming path?

[00:10] Ravi (Candidate): Watermarks. In Flink you define a watermark that says "events more than X seconds late are discarded." A common setting is 30 seconds — events up to 30 seconds late are included in the correct window; events later than that are either dropped or routed to a side output for separate handling. The tradeoff is between data completeness and latency. A larger watermark means more complete windows but more latency in emitting results.

[00:16] Cleo (Interviewer): Now a coding question. You have a PySpark DataFrame with columns user_id, event_type, and event_timestamp. Write a query to find, for each user, the time between their first and last event.

[00:17] Ravi (Candidate): Sure. I'd use groupBy with aggregations.

```python
from pyspark.sql import functions as F

result = df.groupBy("user_id").agg(
    F.min("event_timestamp").alias("first_event"),
    F.max("event_timestamp").alias("last_event")
).withColumn(
    "time_between_secs",
    (F.col("last_event").cast("long") - F.col("first_event").cast("long"))
)
```

This gives seconds between first and last event per user. If event_timestamp is already a timestamp type, the cast to long gives Unix seconds. For millisecond precision I'd divide by 1000 after casting to long with milliseconds.

[00:19] Cleo (Interviewer): What if the DataFrame has 10 billion rows and the groupBy causes a data skew — a few user_ids have 100 million events each while most have under 100?

[00:20] Ravi (Candidate): Skew is a real problem with groupBy in Spark. For this specific aggregation — min and max — one approach is a two-stage aggregation: first aggregate within partitions using mapPartitions to get local min/max, then do a final reduce. That avoids shuffling all 100 million rows for the skewed users to a single executor.

Another approach is salting: add a random suffix to user_id for the first stage, aggregate within salted partitions, then aggregate again on the original user_id. But salting is more useful for operations like count or sum — for min/max it's simpler to just use a partial aggregation approach. In practice I'd also check whether those 100M-event users are legitimate data or a data quality issue — that volume is unusual and might indicate a bot or a logging bug.

After those changes the pipeline went from two to three failures per week down to roughly one per month, and the failures that did occur were caught faster because the error messages were specific.

[00:33] Cleo (Interviewer): Walk me through how you'd handle slowly changing dimensions in a data warehouse. Specifically: a user's location changes — how do you preserve history?

[00:34] Ravi (Candidate): Classic SCD problem. There are three standard approaches. SCD Type 1: overwrite — just update the record. You lose history, which is unacceptable if you need to know where a user was at time of purchase. SCD Type 2: add a new row per change with effective_date and expiration_date columns and a is_current flag. Queries filter on is_current=true for current state, or join on the date range for historical state. This is the most common approach for analytics because it preserves full history. SCD Type 3: add columns like previous_location and changed_at — you get the prior value but only one level of history. SCD Type 2 is almost always the right answer for user location because purchases and events need to be joined to the user's location at that point in time, not their current location.

The implementation tradeoff: SCD Type 2 significantly increases row count — one user with ten location changes becomes eleven rows. You need to carefully design your fact-to-dimension joins to avoid fan-out. In BigQuery or Snowflake I'd use a surrogate key on the dimension row and join facts on that surrogate key rather than the user_id.

Option C: hourly batch with fast query on BigQuery, what we have now, $0 incremental. I ran through what "one second" would actually enable in her workflow versus "five minutes." She thought about it and realized the five-minute latency was fine for her actual use case — she'd assumed one second was necessary but hadn't verified whether analysts actually needed per-second freshness. We shipped Option B. She was happy with the outcome.
