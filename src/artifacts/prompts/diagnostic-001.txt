R-005 (P-105, resume_parse): The new model nested the `contact_info` object one level deeper than expected: `{"candidate": {"contact_info": {...}}}` instead of the flat top-level `{"contact_info": {...}}`. This extra nesting layer caused the downstream parser to fail field resolution.

R-006 (P-106, resume_parse): Both models failed this run. The old model returned `education` as a list of strings rather than a list of structured objects with `{degree, institution, year}`. The new model replicated the same structural error and additionally dropped the `graduation_year` key entirely.

R-007 (P-107, skill_extraction): The new model returned `proficiency_levels` as a dict mapping skill names to strings (`{"Python": "expert"}`), but the schema requires a list of objects: `[{"skill": "Python", "level": "expert"}]`. Type mismatch caused schema validation failure.

R-009 (P-109, skill_extraction): The new model returned `null` for the `certifications` field rather than an empty list `[]`. The schema specifies `certifications: list[str]`, which does not permit null. The old model correctly returned `[]`.

    <pattern_analysis>
The dominant failure pattern across this run set is field type regression: gemini-3.1-flash-lite consistently returns structured fields as simpler types than the schema requires. Arrays are collapsed to strings (R-001, R-015, R-029), lists of objects are collapsed to flat dicts or lists of strings (R-007, R-011, R-013, R-027), and nullable fields are incorrectly returned as null instead of empty collections (R-009). This pattern spans all five task families — candidate_eval, resume_parse, skill_extraction, offer_gen, and doc_classification — indicating a model-level behavioral shift rather than a prompt-specific issue.

A secondary pattern is missing required fields and structural nesting errors. R-003, R-020, and R-025 show cases where the new model drops a required key entirely, while R-005 and R-018 add unexpected nesting layers or restructure the field hierarchy. R-006 demonstrates both models sharing a structural misunderstanding, which predates the migration. These structural failures are distinct from the type-collapse failures and suggest the new model has weaker adherence to field presence constraints than the old model.

The new task families (offer_gen, doc_classification) added in the expanded battery show the same failure distribution as the original three families. Offer_gen exhibits both structural flattening (R-016, R-018) and field omission (R-020). Doc_classification shows type coercion (R-022) and field omission (R-025). This cross-family consistency confirms the failure mechanism is model-level, not prompt-level. Regressions are not concentrated in any single task family but distributed uniformly, ruling out prompt-specific causes as the root issue.

Excluding R-006 and R-023 (pre-existing failures), the new model introduces 15 net regressions on prompts the old model handled correctly, out of 28 previously passing runs. Regression rate: 53.6%. This exceeds the 50% threshold for block_migration. The task families break down as: candidate_eval (4 regressions), resume_parse (4 regressions), skill_extraction (4 regressions), offer_gen (3 regressions), doc_classification (2 regressions). This uniform distribution across all five families rules out prompt-level causes.
