[00:03] Nadia (Interviewer): Per-user limits, 500K RPS aggregate across all 50 servers, so about 10K RPS per server on average.

[00:04] Alex (Candidate): Got it. The core challenge is that with 50 servers, a local counter per server would let users exceed their limit by up to 50x if they spread requests evenly. We need a shared state layer. Options: a centralized counter store like Redis, or a token-bucket gossip protocol where servers share counts periodically.

I'd go with Redis in cluster mode with a sliding window log algorithm. Here's the tradeoff: sliding window is more accurate than fixed window — it avoids the boundary burst problem — but it uses more memory because we store timestamps per request. For a user with a limit of 1,000 requests per minute, we store up to 1,000 timestamps. At 500K RPS with say 100,000 active users, that's manageable.

The Redis implementation: for each request I do a Lua script atomically — ZADD the current timestamp, ZREMRANGEBYSCORE to drop entries older than the window, ZCARD to get the count, then check against the limit. One round-trip. At 50 servers each doing 10K RPS, Redis gets 500K ops/second. Redis can handle about 1 million ops/second on a single node, so we're fine with headroom. For the 5ms latency budget: Redis round-trip within the same datacenter is typically under 1ms, so we're well within budget.

For failure mode: if Redis is unavailable, I'd fail open with a local fallback counter that's conservative — say 50% of the normal limit — rather than blocking all traffic. The 5% accuracy requirement relaxes during the outage window.

[00:09] Nadia (Interviewer): What if a single user suddenly sends 100K RPS — 20% of total capacity — trying to overwhelm your Redis layer?

[00:13] Nadia (Interviewer): Let's switch to a coding problem. Given a string, find the length of the longest substring without repeating characters.

[00:14] Alex (Candidate): Classic sliding window. I'll use a hashmap to track the last seen index of each character, and two pointers for the window boundaries. When I see a character I've seen before within the current window, I move the left pointer to the character after its last occurrence.

```python
def length_of_longest_substring(s: str) -> int:
    last_seen = {}
    left = 0
    max_len = 0
    for right, char in enumerate(s):
        if char in last_seen and last_seen[char] >= left:
            left = last_seen[char] + 1
        last_seen[char] = right
        max_len = max(max_len, right - left + 1)
    return max_len
```

Time O(n), space O(min(n, k)) where k is the character set size — for ASCII that's a constant 128 so effectively O(1) space.

[00:21] Alex (Candidate): About ten months ago, we had an intermittent latency spike on our search API — P99 latency was jumping from 80ms to 2,400ms for about 90 seconds every 15 minutes, then recovering. I correlated timestamps: the latency spikes aligned within 2 seconds of the job start. Confirmed causation by temporarily disabling the job — spikes stopped immediately.

Root cause: the cache rebuild job was doing a full table scan on a 40-million-row table without a covering index, locking rows in a way that blocked concurrent search queries. The fix had two parts. [00:30] Nadia (Interviewer): Walk me through how you'd design a job queue system for background tasks. Assume you need exactly-once semantics and you're handling roughly 10,000 tasks per minute.

[00:31] Alex (Candidate): Exactly-once is the hard part. I'd build this on top of a transactional database combined with a queue. The pattern: when a client wants to enqueue a task, they INSERT a row into a `jobs` table with status="pending" inside a database transaction — that INSERT is atomic and durable. A background worker polls the table using SELECT FOR UPDATE SKIP LOCKED — this atomically locks a row for one worker without blocking others. The worker processes the job, then UPDATE status="completed" in the same transaction. If the worker crashes mid-processing, the transaction rolls back and the job goes back to "pending." SELECT FOR UPDATE SKIP LOCKED ensures no two workers grab the same row.

The tradeoff versus a dedicated queue like Kafka or SQS: the database approach has better exactly-once guarantees because you're using ACID transactions, but it doesn't scale as well for very high throughput. At 10,000 tasks per minute — about 167 per second — a well-tuned Postgres with proper indexing can handle that easily. If you needed 10,000 per second, you'd move to a dedicated queue.

[00:36] Nadia (Interviewer): What's your approach to database migration strategy in a high-traffic system? You need to add a NOT NULL column to a table with 50 million rows.

[00:37] Alex (Candidate): Never do it in one step on a production table that size. The naive approach — ALTER TABLE ADD COLUMN NOT NULL — blocks the table for potentially minutes on a 50M row table, which is a production outage. The safe approach is three steps. Step one: ADD COLUMN with a DEFAULT value — this is instant on Postgres 11+ because it just updates the catalog, not the rows. Step two: backfill in batches — UPDATE the new column in chunks of 10,000 rows with a sleep between batches to avoid overwhelming I/O. Step three: once backfill is complete, add the NOT NULL constraint as a separate operation. In Postgres, you can add a constraint as NOT VALID, which skips existing rows, then VALIDATE CONSTRAINT in a later deploy, which takes only a share lock instead of a table lock. We did this at my previous company for a column we added to our `transactions` table — the table had about 60 million rows and we completed the migration with zero downtime and less than 5% I/O overhead during the backfill phase.
